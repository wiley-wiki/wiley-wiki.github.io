事务处理是保证多个SQL操作的一致性，如果执行一条SQL语句则全部SQL语句失效

# 存储引擎与提交模式

# 查看MySQL支持的存储引擎

show engines;

默认使用的是 InnoDB 存储引擎

若要修改存储引擎，则可使用下面的SQL语句进行修改

ALTER TABLE stu ENGINE=InnoDB;

# 提交模式

MySQL的提交模式默认是自动提交，发送一条执行一条，会立即看到结果

# 事务提交

# 开启事务

START TRANSACTION 或者 BEGIN 语句都可以显式开启一项事务

# 事务回滚

ROLLBACK 事务回滚，会结束用户的事务，并撤销正在进行的所有未提交的修改

# 提交事务

COMMIT 事务提交，并使已对数据库的所有修改成为永久性的

# 全局开启或关闭自带提交模式

# 开启自动提交

SET AUTOCOMMIT = 1;
1

# 关闭自动提交

SET AUTOCOMMIT = 0;
1

全局事务提交，则所有的SQL语句执行后，都需要进行 COMMIT 事务提交

# 事务提交案例

# 事务提交

START TRANSACTION;
INSERT INTO stu2(sname,class_id,sex) VALUES('王五',3,1);
COMMIT;

只有在 COMMIT 之后，才会将所有的修改永久性的到数据库

事务语句必须在开启事务和提交事务语句之间，也可在事务回滚语句之前。提交事务或事务回滚之后的SQL语句则不是事务语句。

# 事务回滚

BEGIN;
INSERT INTO stu2(sname,class_id,sex) VALUES('李兰',2,1);
ROLLBACK;

当进行事务回滚，则不会插入数据

# 事务隔离级别

# 隔离级别

隔离级别 是否脏读 是否不可重复读 是否幻读 说明
读未提及(read-uncommitted) 是 是 是 事务还没提交，修改的数据可被其他事务读取到
不可重复读(read-committed) 否 是 是 其他事务不能读取到事务未提交的数据，事务提交后才能被其他事务读取
可重复读(repeatable-read) 否 否 是 多次读取修改和查询的数据会返回第一次查询的快照，即便其他事务已经提交修改数据。事务在执行期间看到的数据必须是一致的
串行化(serializable) 否 否 否 事务最可靠隔离级别，可避免脏读、不可重复读、幻读情况的发生

# 查询隔离级别

select @@tx_isolation;
1

# 设置隔离级别

set session transaction isolation level read uncommitted;
1

# 读未提交（脏读）

脏读就是事务没有提交，但是未提交的数据可被其他事务读取到

事务A设置 read-committed 隔离级别，开启事务修改数据未提交，事务B设置 read-committed 隔离级别，开启事务查询数据，能读取到事务A没有提交的数据

# 可重复读

可重复读，所有事务在执行期间看到的数据都是一致的。

事务A设置 repeatable-read 隔离级别，开启事务修改数据并提交，事务B第一次读取数据会读取到事务A修改的数据，事务A第二次修改数据并提交，事务B和事务A查询的数据都是事务A第一次修改的数据

# 幻读

幻读，并不是说两次读取获取的结果集不同，实际情况是：select 某记录是否存在，不存在，准备插入此记录，但执行 insert 时发现此记录已存在，无法插入，此时就发生了幻读。
